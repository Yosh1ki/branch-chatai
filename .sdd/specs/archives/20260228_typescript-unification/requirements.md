# 要件定義書

## 機能概要
`.js` と `.ts` の重複実装を整理し、`conversation-history` と `usage-limiter` を含む対象機能を TypeScript に統一する。これにより、実装の二重管理を解消し、チャットAPI周辺の保守性と一貫性を向上させる。

## ユーザーストーリー
- 開発者として、重複している JavaScript/TypeScript 実装を TypeScript に統一したい。なぜなら同一ロジックの多重保守をなくし、変更時の不整合リスクを減らしたいから。

## 機能要件
### 要件1：重複実装の棚卸しと統合方針の確定
- `.js` と `.ts` が併存する対象モジュールを特定し、TypeScript 側へ統一する対象と移行方針を明確化する。
- 受入基準：
  - [ ] `conversation-history` と `usage-limiter` を含む対象ファイルの対応表（移行元/移行先）が定義されること
  - [ ] 統一後に参照される正規実装が TypeScript 側で一意に決まっていること

### 要件2：TypeScript への統一実装
- 対象ロジックを TypeScript 実装へ統合し、既存の呼び出し側が TypeScript 実装を参照するよう更新する。
- 受入基準：
  - [ ] 対象機能で JavaScript 実装への依存が解消されていること
  - [ ] 統合後も既存仕様と同等の振る舞いを維持していること

### 要件3：品質確認と回帰防止
- 変更対象に対して静的検証・動作検証を実施し、統一による回帰がないことを確認する。
- 受入基準：
  - [ ] 関連する lint/typecheck/test が成功すること
  - [ ] 主要フロー（会話履歴管理、利用制限判定）で回帰が確認されないこと

## 非機能要件
- パフォーマンス：TypeScript 統一により API 応答時間や制限判定の処理時間を悪化させないこと。
- セキュリティ：認証済みユーザーに対する履歴参照・利用制限判定のアクセス制御を変更しないこと。
