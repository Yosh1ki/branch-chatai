# 要件定義書

## 機能概要
日次メッセージ上限（Freeプラン10件/日）のリセット時刻を仕様として明確化し、実装上も同一タイムゾーン基準で判定することで、日付境界の揺れによる上限判定の不整合を防ぐ。

## ユーザーストーリー
- ユーザーとして、日次上限のリセット時刻を一貫した基準で扱ってほしい。なぜなら利用可能件数が想定どおりに回復し、安心してチャットを継続できるから。
- 運用者として、日次上限判定の基準時刻を明文化したい。なぜなら仕様と実装のずれを防ぎ、問い合わせ対応を簡単にできるから。

## 機能要件
### 要件1：日次上限リセット時刻の仕様明文化
- Freeプランの日次上限リセット時刻を単一タイムゾーンで定義し、仕様書に明記する。
- 受入基準：
  - [ ] 日次上限のリセット時刻と基準タイムゾーンが要件として明示されていること
  - [ ] 基準タイムゾーン以外のローカル時刻を判定基準にしない方針が記載されていること

### 要件2：上限判定ロジックのタイムゾーン固定
- 日次上限チェックで使用する日付境界計算を、仕様で定義した基準タイムゾーンに固定する。
- 受入基準：
  - [ ] 日次上限チェック処理が固定タイムゾーンの境界で日付判定すること
  - [ ] サーバー実行環境のローカルタイムゾーンに依存しないこと

### 要件3：境界時刻の動作保証
- リセット直前・直後の境界時刻で、上限超過判定と件数リセットが期待どおりに動作することを確認可能にする。
- 受入基準：
  - [ ] リセット直前では当日件数が維持されること
  - [ ] リセット直後では日次件数が新しい日として扱われること

## 非機能要件
- パフォーマンス：日次上限判定の追加処理によるAPI応答劣化を実運用で無視できる範囲に抑えること。
- セキュリティ：上限判定の基準時刻・タイムゾーン設定が外部入力で改変されないこと。
